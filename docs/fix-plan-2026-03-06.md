# ImmichTo115 修复拆分计划（基于 2026-03-06 审查报告）

## 说明

- 本文档用于把 `docs/code-review-2026-03-06.md` 中的问题拆成可执行批次。
- 当前仍然是**计划阶段**，本文不包含代码修复。
- 拆分原则：
  1. 先处理高风险、会影响安全与数据可靠性的问题。
  2. 再处理会影响部署稳定性与用户认知的问题。
  3. 最后处理文档、可访问性与维护性收尾项。

## 总体顺序

建议按以下 5 个批次推进：

1. **批次 A：认证边界与首次配置保护**
2. **批次 B：备份执行结果与通知可信度**
3. **批次 C：恢复能力边界与前端异常状态**
4. **批次 D：部署运行时与发布链路收口**
5. **批次 E：文档一致性、可访问性与维护性清理**

---

## 批次 A：认证边界与首次配置保护

### 目标
修复“开启访问保护但初始化未完成时整体失效”的问题，确保管理入口、API、WebSocket 的保护边界清晰且一致。

### 包含问题
- 审查项 1：初始化未完成时，访问保护整体失效
- 审查项 4：路由守卫在系统状态接口失败时直接放行

### 涉及文件
- `internal/api/router.go`
- `internal/config/config.go`
- `web/src/router.ts`
- `web/src/api.ts`
- `web/src/views/Settings.vue`
- `web/src/views/Wizard.vue`

### 预期结果
- setup 前后有明确的认证边界
- 前端不会在“状态未知”时直接放行进入管理页
- 访问保护开启后的行为与 UI 文案一致

### 风险
- 这是全站入口级改动，容易影响首次安装流程
- 需要同时验证无鉴权、新开启鉴权、已有鉴权三种路径

### 验证建议
- 首次启动未配置状态访问 `/`、`/api/v1/config`、`/ws/logs`
- 开启访问保护后刷新页面，确认浏览器立即进入认证挑战
- 鉴权失败、401、后端不可达三种情况下的前端路由行为

---

## 批次 B：备份执行结果与通知可信度

### 目标
保证备份任务的“成功/失败/停止”状态能够真实反映 `rclone` 执行结果，不再出现日志失败但整体被当成成功的情况。

### 包含问题
- 审查项 2：备份失败可能不会改变整体成功结论
- 审查项 9：Bark 通知 URL 构造缺少转义
- 审查项 10：Dashboard 缺少清晰的“连接已降级”状态表达（与任务状态感知相关）

### 涉及文件
- `internal/rclone/rclone.go`
- `internal/api/router.go`
- `internal/notify/bark.go`
- `internal/api/ws.go`
- `web/src/views/Dashboard.vue`
- `web/src/api.ts`

### 预期结果
- `rclone` 的退出状态能结构化传回上层
- 备份任务只在真实成功时才发送成功通知
- 停止任务与失败任务在日志、状态、通知中有明确区分
- Dashboard 能明确显示“日志断开 / 状态过期 / 备份失败”

### 风险
- 会影响备份主流程与通知逻辑，需避免引入新的任务卡死或假失败

### 验证建议
- 手动构造失败备份（错误路径/错误远端）验证状态与通知
- 手动停止备份验证收尾行为
- Bark 标题/正文包含中文、空格、emoji 场景
- Dashboard 断开 WebSocket / 轮询失败时的 UI 表现

---

## 批次 C：恢复能力边界与前端异常状态

### 目标
让“已实现的能力”和“尚未实现的能力”在前端、后端、README 中完全对齐，并修正 Restore Explorer 的误导性交互。

### 包含问题
- 审查项 3：恢复功能在 UI/README 中被描述得比实际能力更完整
- 审查项 11：Restore Explorer 的选择语义不完整
- 审查项 13：Layout 的“服务运行中”文案不是实时健康状态

### 涉及文件
- `web/src/views/RestoreExplorer.vue`
- `web/src/components/Layout.vue`
- `README.md`
- 如决定真正补恢复能力，还会涉及 `internal/api/router.go`

### 预期结果
- Restore Explorer 当前能力边界清晰：浏览、选择、恢复、下载分别是什么状态
- 顶部全局状态与真实服务状态更接近
- README 不再把“浏览能力”和“恢复能力”混写

### 风险
- 这是用户感知最直接的一批改动，容易影响当前宣传口径

### 验证建议
- 空目录、有文件、文件夹混合三种 Restore Explorer 场景
- 文件夹选择与文件选择语义验证
- README 与 UI 文案逐项对照

## 批次 D：部署运行时与发布链路收口

### 目标
解决部署默认值、容器健康检查、安装脚本与发布链路中不一致或高风险的部分，提升可运维性。

### 包含问题
- 审查项 6：Docker Compose 健康检查与镜像依赖不一致
- 审查项 7：默认以 root 运行，放大运行时风险
- 审查项 8：配置目录权限实现与 README 建议不一致
- 审查项 15：安装脚本与版本展示能力存在漂移迹象

### 涉及文件
- `deploy/Dockerfile`
- `deploy/docker-compose.yml`
- `deploy/install.sh`
- `deploy/uninstall.sh`
- `cmd/server/main.go`
- `internal/config/config.go`
- `.github/workflows/release.yml`

### 预期结果
- Compose 健康检查可真实反映容器状态
- 服务与容器默认运行权限更合理
- 配置目录权限与 README 建议一致
- 安装脚本的升级信息与程序版本能力对齐
- 发布脚本、程序版本、README 三者一致

### 风险
- 会直接影响 Docker / systemd 现有用户的升级路径
- root 降权需要重新审视挂载目录与文件权限

### 验证建议
- Docker Compose 冷启动 / 重启 / 健康检查
- install / update / uninstall 全链路验证
- Release tag 构建、二进制版本输出、安装脚本版本展示一致性

## 建议执行顺序与原因

### 第一步：批次 A + 批次 B
这是最值得先做的组合：
- 批次 A 解决安全边界
- 批次 B 解决数据可靠性

这两组都直接影响“能不能安全用”和“备份到底可不可信”。

### 第二步：批次 D
认证和备份可信度稳定后，再收口部署与发布链路，避免“修好了代码，但交付还是不稳定”。

### 第三步：批次 C
这批更多影响用户认知与功能边界，适合在后端主链稳定后统一处理。

### 第四步：批次 E
最后再做重复实现、文档与可访问性收尾，避免在高优先级修复期间引入不必要的大范围重构。

---

## 建议的工作单拆分

可以直接按下面方式建任务：

1. `AUTH-01`：收紧 setup 前后的认证边界
2. `AUTH-02`：修正前端状态未知时的路由放行逻辑
3. `BACKUP-01`：让 rclone 失败可结构化上报到备份主流程
4. `BACKUP-02`：通知结果与真实备份结论对齐
5. `OPS-01`：修复 Docker 健康检查与镜像依赖不一致
6. `OPS-02`：收口 root 运行与配置目录权限
7. `RELEASE-01`：统一版本能力、安装脚本与发布流程
8. `RESTORE-01`：明确 Restore Explorer 当前能力边界
9. `DOC-01`：修正文档/API 漂移
10. `UI-01`：抽离 Settings / Wizard 公共配置逻辑
11. `A11Y-01`：补齐 toast 与关键交互控件语义

---

## 结论

这份拆分计划的核心目标是：
- **先保安全与数据可信度**
- **再保部署与发布稳定性**
- **最后收用户体验与维护性**

如果进入下一步实施，最推荐从 **批次 A（认证）** 和 **批次 B（备份结果）** 开始。
